# storix 0.2.0 — Hexagonal Architecture Design

- **Date:** 2026-07-10
- **Status:** Approved for planning
- **Release:** 0.2.0 (breaking; semver 0.x)

## 1. Motivation

Today every provider implements the entire ~30-method unix interface, and the whole
tree is duplicated for async:

| File | Lines | Content |
|---|---|---|
| `providers/azure.py` | 612 | sync Azure, full unix interface |
| `aio/providers/azure.py` | 652 | same, async |
| `providers/local.py` | 305 | sync local, full unix interface |
| `aio/providers/local.py` | 334 | same, async |

~1,900 lines of provider code where most logic is not provider-specific. Unix semantics
(parent checks in `touch`, `cd` validation, hidden-file filtering, error behavior) are
hand-written four times and have already drifted (e.g. sync vs async local `__init__`
path resolution). Error semantics are inconsistent: `cat`/`cd`/`mv` raise while
`touch`/`rm`/`rmdir`/`echo` log and return `False`.

**Drivers (in priority order):** behavioral consistency across providers and flavors;
clean architecture while the project is small. Secondary wins: cheap new providers,
sync/async dedup.

**Positioning context:** storix competes as the *ergonomic, sandboxed filesystem layer
for humans and AI agents* — not as plumbing against fsspec/opendal/obstore. This
refactor is the foundation for that layer regardless of adoption outcome: it cuts
maintenance cost ~70% even as a personal tool.

## 2. Decisions

| Decision | Choice |
|---|---|
| API compatibility | Full redesign freedom; ships as breaking 0.2.0 |
| Sync/async | **Async-first codegen**: hand-write `_async/`, generate `_sync/` (httpcore/psycopg3 pattern); tests generated too |
| Port shape | **Wider port** (~12 required methods incl. `move`/`copy`/`delete_tree`/`du`) + generic fallback helpers; `BackendBase` ABC pre-wires fallbacks |
| Error semantics | **Raise everywhere** — typed `storix.errors` exceptions; no log-and-return-`False` |
| Wiring | **Explicit composition** `Storix(backend)` + settings-driven `get_storage()` factory |
| Cross-cutting concerns | **Layers** — middleware backends implementing the same port (`SandboxLayer` ships now) |
| Asymmetric capabilities | Capability flags per backend + `UnsupportedOperationError`; containers are backend config, never path data |

## 3. Package layout & codegen

```
src/storix/
├── _async/                  # SOURCE OF TRUTH — hand-written
│   ├── core.py              #   class Storix: all unix semantics
│   ├── factory.py           #   get_storage()
│   └── backends/
│       ├── _proto.py        #   StorageBackend protocol + BackendBase ABC
│       ├── generic.py       #   fallbacks: move=copy+delete, du=walk, ...
│       ├── local.py         #   LocalBackend (aiofiles)      [hand-written]
│       ├── azure.py         #   AzureBackend (azure .aio SDK) [hand-written]
│       └── memory.py        #   MemoryBackend (dict-based)    [hand-written]
├── _sync/                   # GENERATED by scripts/unasync.py — never hand-edited
│   └── (mirror of _async; backends hand-written, see §3.2)
├── aio/__init__.py          # re-exports from _async
├── __init__.py              # re-exports from _sync
├── layers/                  # SandboxLayer (port middleware)
└── shared, IO-free: types.py, models.py, errors.py, pathops.py,
    settings.py, constants.py, core/ (tree/find/wc), utils/
tests/
├── _async/                  # hand-written async tests (source of truth)
└── _sync/                   # GENERATED sync tests
scripts/unasync.py           # token-substitution codegen
```

### 3.1 Codegen safety (no trust in review)

1. **Whitelist-driven**: the script hard-fails on any async construct without a
   registered translation. `asyncio.gather`, `TaskGroup`, bare `asyncio.` usage in
   `_async/` are build errors, never silently-wrong sync output.
2. **Single concurrency shim**: core concurrency only via
   `storix._compat.gather(*aws, limit=...)`; its registered sync translation is a
   sequential loop. Concurrency semantics differ in exactly one audited function.
3. **Generated tests**: async tests are the source; the sync suite is generated, so
   generated code is verified by generated tests.
4. **CI**: regenerate + `git diff --exit-code`; hand-edits to `_sync/` fail the build.

### 3.2 What is hand-written twice

Backends only. The local backend's `aiofiles`-vs-`open` and Azure's `.aio`-vs-sync SDK
differences make substitution rules brittle; at ~130 lines each the duplication is
small and the conformance suite (§8) is the drift guard. Core, factory, protocol,
generic helpers, and tests are generated.

## 4. The port

```python
class StorageBackend(Protocol):
    capabilities: Capabilities   # frozen flags, see §7

    async def read(self, path: PurePosixPath) -> bytes: ...
    def read_stream(self, path) -> AsyncIterator[bytes]: ...
    async def write(self, path, data: AsyncIterator[bytes], *,
                    mode: Literal['w', 'a'], content_type: str | None) -> None: ...
    async def delete(self, path) -> None: ...
    def list_dir(self, path) -> AsyncIterator[Entry]: ...   # Entry(name, is_dir)
    async def stat(self, path) -> RawStat: ...               # size, times, kind
    async def exists(self, path) -> bool: ...
    async def make_dir(self, path, *, parents: bool) -> None: ...
    # native-fast-path ops — required; generic fallbacks one import away
    async def move(self, src, dst) -> None: ...
    async def copy(self, src, dst) -> None: ...
    async def delete_tree(self, path) -> None: ...
    async def du(self, path) -> int: ...
```

- **Streaming is the substrate**: `read_stream` and iterator-based `write`/`list_dir`
  ship in the port from day one (retrofitting streaming into a port contract is
  breaking; future `fs.open()`, progress callbacks, and large-dir listing all need it).
- **`BackendBase` ABC** pre-wires generic fallbacks (`move` = copy+delete, `du` = walk,
  `read` = collect `read_stream`, ...). Providers subclass it and override what they
  can do natively. Consequence: future port growth (`delete_many`, `copy_many`) is a
  new method + generic default — **non-breaking for third-party backends**.
- **Path contract**: backends receive fully-resolved absolute POSIX paths. The core has
  already applied cwd, `~`, `..`; layers have applied sandbox translation. Backends do
  zero path logic beyond mapping to native form (local: `Path(p)`; Azure:
  container-relative string).
- **Containers are backend config**: `AzureBackend(container='raw')` fixes the
  namespace at construction. Paths never contain container names;
  `allow_container_name_in_paths` is removed. Cross-container ops are out of scope
  (two backends = two filesystems).
- **Error contract**: backends translate native failures (`OSError`,
  `HttpResponseError`) into `storix.errors` at the boundary. The core only sees storix
  errors. Taxonomy: `StorageError` base; `PathNotFoundError`, `NotADirectoryError`,
  `IsADirectoryError`, `PermissionDeniedError`, `AlreadyExistsError`,
  `UnsupportedOperationError`.

## 5. The core: `class Storix`

One class owning everything unix: cwd/home state, `chroot`, path resolution (today's
`PathLogicMixin` becomes pure functions in `pathops.py`), and every command — `ls`
(hidden filter, abs/rel), `touch` (parent checks), `cat`, `echo` (stream normalization
+ chunking), `cd` (is-a-directory validation), `mv`/`cp`/`rm`/`rmdir`,
`tree`/`find`/`du`/`stat`/`wc`, `exists`/`isdir`/`isfile`/`empty`, `make_data_url`,
context manager. Every failure raises. Estimated ~500–600 async lines replacing ~1,900
provider lines; sync generated for free.

`BaseStorage`, `_base_tree.py`, the old `Storage` protocol in `providers/_proto.py`,
and all per-provider unix implementations are deleted.

## 6. Layers (port middleware)

A layer is a backend that wraps a backend — same protocol, composable:

```python
Storix(SandboxLayer(AzureBackend(container='raw'), root='/tenants/42'))
Storix(backend, sandboxed=True)   # sugar: applies SandboxLayer at backend root
```

**`SandboxLayer`** (ships in 0.2.0) replaces `sandbox.py`'s handler + the
`if self._sandbox` branches scattered through every provider method. One choke point:
normalize the already-resolved virtual path, prefix with `root`, assert the result
stays under `root` (defense in depth). Testable in isolation against `MemoryBackend`.

Future layers use the same shape (backlog): `MetadataLayer` (sidecar metadata store
that upgrades advertised capabilities of capability-poor backends — the local-metadata
answer), `CacheLayer`, `ObservabilityLayer` (audit/progress events for the agent
story), retry.

## 7. Capabilities

Each backend exposes a frozen `Capabilities` flags object (`content_type`,
`custom_metadata`, `presigned_urls`, ...). The core consults it and raises
`UnsupportedOperationError` when an argument can't be honored (explicit failure over
silent dropping). Read-side asymmetry lives in typed stat extensions
(`AzureFileProperties`). `fs.backend` is public as the escape hatch for
provider-specific operations. Layers may upgrade the capabilities of what they wrap.

## 8. Testing

- **Conformance suite** (the centerpiece): one parametrized behavior spec — unix
  semantics *and* error types — run against every backend × flavor combo: memory,
  local, local-via-`temporary()`, Azure (mocked/Azurite), in async (source) and sync
  (generated). This replaces the four provider test files and *is* the enforcement
  mechanism for "one behavior everywhere", including hand-written backend parity.
- **Core unit tests** against `MemoryBackend` (fast, no disk).
- **Per-backend quirk tests** stay small (SAS URLs, container config).
- **`storix.temporary()`**: context manager (sync + async) yielding a sandboxed
  `Storix` over a `mkdtemp`-rooted `LocalBackend`, cleaned up on exit. Public API and
  test utility.

## 9. Public API

```python
from storix import Storix, get_storage
from storix.backends.azure import AzureBackend

fs = Storix(AzureBackend(container='raw'), sandboxed=True)   # explicit composition
fs = get_storage()                                            # settings-driven
from storix.aio import Storix                                 # same names, async

with storix.temporary() as fs: ...                            # scratch filesystem
```

Re-export convention: `storix.backends.*` exposes the `_sync` backends and
`storix.aio.backends.*` the `_async` ones, mirroring `Storix` itself — users never
import from `_async`/`_sync` directly.

### Breaking changes (release notes get a migration table)

| 0.1.x | 0.2.0 |
|---|---|
| `LocalFilesystem(...)` | `Storix(LocalBackend(...))` |
| `AzureDataLake(...)` | `Storix(AzureBackend(...))` |
| `touch`/`rm`/`rmdir`/`echo` return `False` on failure | raise typed errors |
| `allow_container_name_in_paths` | removed (containers are backend config) |
| `sandbox_handler=` | removed (`sandboxed=True` sugar or explicit `SandboxLayer`) |

Surviving unchanged: `get_storage()`, `StorixPath`, `storix.errors` (extended),
method names/unix vocabulary, CLI/shell behavior (internally constructs `Storix`),
settings/env configuration.

## 10. Out of scope for 0.2.0 (agreed backlog, value-ordered)

- **0.2.x — polish:** `echo(atomic=True)` (write-temp-then-`move`), `glob`,
  `progress=` callback (chunk counting over the stream substrate), docs site with
  honest comparison page.
- **0.3.0 — adoption:** `storix.pathlike` pathlib-style adapter (second driving
  adapter over the same core), `FsspecBackend` (provider breadth for free), URL
  factory + `storix.backends` entry-point plugin discovery.
- **0.4.0 — differentiators:** agent story (`ObservabilityLayer` audit events,
  permission layer, possible MCP server), `MetadataLayer`, `CacheLayer`, staged
  transactions (`with fs.transaction():` — stage writes, `move` on commit; documented
  honestly as all-or-nothing *for new writes only*), `delete_many` batch op.

Explicitly rejected: competing with fsspec/opendal on native backend breadth or raw
performance; general rollback transactions (impossible on object stores); silent
degradation of unsupported capabilities.
